Quick Introduction of Batch Normalization

之前已经说过，我们不能直接修改error surface的landscape，error surface如果是很崎岖的时候，它是比较难train的，我们要想能不能直接将山和沟铲平填平，让这个model变得好train一点。

Batch Normalization 就是一种把山铲平的想法

优化问题是一个十分现实的问题，有时候就算我们的error surface是convex的，是一个凸的形状，都不一定很好train

假设有两个参数，对Loss的斜率差别非常大，在w1w_1w1​这个方向上，你的斜率变化很小，在w2w_2w2​这个方向上斜率变化很大

如果是固定的learning rate，你可能很难得到很好的结果，所以我们才说你需要adaptive的learning rate，比如Adam等等比较进阶的optimization的方法，就可以得到比一般的gradient descent更好的结果。

而现在我们要从另一个方向想，直接把难做的error surface 改掉，看看能不能够改的好做一些呢？

w1w_1w1​跟w2w_2w2​之间的变换 斜率的改变差很多的这种状况是哪里来的呢？

假设我们现在有一个非常简单的model，它的输入是x1x_1x1​跟x2x_2x2​，它对应的参数就分别是w1w_1w1​和w2w_2w2​，它是一个linear的model，没有activation function

w1w_1w1​乘x1x_1x1​，w2w_2w2​乘x2x_2x2​加上bbb以后就得到yyy，然后会计算 y 跟 y^\hat{y}y^​ 之间的差距当做 e,把所有 training data e 加起来就是你的 Loss，然后去 minimize 你的 Loss

那么什么情况会发生这种不好train的error surface呢？

当我们对w1w_1w1​有一个个小小的改变，比如加上了一个δw1\delta w_1δw1​的时候，这个L也会有一个改变，那这个w1w_1w1​呢，是透过w1w_1w1​改变的时候，你就改变了yyy，yyy改变的时候你就改变了eee，接下来就改变了LLL，那么什么时候w1w_1w1​的改变会对LLL的影响很小呢？也就是它在error surface上的斜率会很小呢？一个可能性是当你的input很小的时候，假设x1x_1x1​的值在不同的training example里面，它的值都很小，当我们乘上w1w1w1之后影响当然也就很小了。

反之，如果今天x2x_2x2​的值很大，当你的w2w_2w2​有一个小小的变化的时候，虽然w2w_2w2​的变化可能很小，但是因为它乘上了x2x_2x2​，x2x_2x2​的值很大，那对于yyy的变化也就很大了，那eee的变化就很大，LLL的变化就很大，就会导致我们在www这个方向上，做变化的时候，我们把www改变一点，那我们的error surface就会有很大的变化。

所以我们可以发现说，既然这个linear的model当我们input的feature，每一个dimension的值，它的scale的差距很大，就可能产生这种error surface，不同的方向上，斜率非常的不同，坡度非常的不同。

所以怎么办？我们能不能给feature内的不同dimension，让他们有同样数值的范围。

如果可以给不同的dimension，同一个数值范围，就可以作出比较好的error surface

这其实是有很多不同的方法可以实现的，统称为Feature Normalization

Feature Normalization

这里要说的只是Feature Normalization的一种可能性，它并不是Feature Normalization的全部，假设x1x^1x1到xRx^RxR是我们所有的训练资料的feature vector

我们将不同资料也就是不同feature vector，同一个维度的数值，都取出来计算某一个维度iii的mean为mim_imi​，计算第iii个dimension的standard deviation，使用σi\sigma_iσi​表示

接下来做一种normalization，这种normalization其实就叫标准化，其实就是standardization，这边为了方便统称normalization了。

将这边的某一个数值xxx，减掉这一个dimension算出来的mean，再除掉standard deviation，得到新的数值，再把这个新的数值塞回去，以下都用tilde代替normalize后的数值。那么做完normalize后有什么好处呢？

  • 做完normalize之后，这个dimension上面的数值就会平均是0，然后他的方差就会是1，所以这一排数值的分布就会在0上下
  • 对每一个dimension都做一样的normalization，就会发现所有feature不同dimension的数值都在0上下，那你就可能制造一个，比较好的error surface

这样就满足了我们上面所提到的把error surface“铲平”去让训练过程更加顺利些。

Considering Deep Learning

x~\tilde{x}x~ 代表 normalize 的 feature,把它丢到 deep network 裡面,去做接下来的计算和训练,所以把 x1x_1x1​ tilde 通过第一个 layer 得到 z1z^1z1,那你有可能通过 activation function,不管是选 Sigmoid 或者 ReLU 都可以,然后再得到 a1a^1a1,然后再通过下一层等等,那就看你有几层 network 你就做多少的运算

这边的a1a3a^1 a^3a1a3这边的z1z3z^1 z^3z1z3，其实也是另一种input，这边x~\tilde{x}x~虽然已经做了normalize了，但是通过w1w_1w1​以后它就没有做normalize，如果x~\tilde{x}x~通过w1w_1w1​得到是z1z_1z1​,而z1z_1z1​不同的dimension之间，它的数值的分布仍然有很大的差异，那我们要训练w2w_2w2​第二层的参数会不会也有困难呢？

对于w2w_2w2​来说，这边的aaa或者说zzz也是一种feature，只是再hidden layer里面的feature，我们应该对这些hidden layer中的feature也做normalization。

那么如果选择的是Sigmoid的话，那可能比较推荐对zzz做Feature Normalization，因为Sigmoid是S形的，那它在000附近的斜率会比较大，所以如果你对zzz做Feature Normalization，把所有的值都挪到000附近，那么你到时算gradient descent的时候算出的值会比较大。

其实如果不是用Sigmoid，用其他的激活函数的话，选aaa做Normalization的结果可能会更好也说不定。

这个Normalization的工作放在激活函数前或后都是可以的，在实作上，可能没有太大的差别

怎么做对zzz做Feature Normalization

我们可以把zzz，想成是另外一种feature，我们这边有z1z2z3z^1 z^2 z^3z1z2z3，我们就把z1z2z3z^1 z^2 z^3z1z2z3拿出来

  • 算一下它的mean，这边μ\muμ是一个vector，我们就把 z1z^1z1 z2z^2z2 z3z^3z3,这三个 vector 呢,把它平均起来,得到 μμμ 这个 vector
  • 算一个 standard deviation,这个 standard deviation 呢,这边这个成 σ\sigmaσ,它也代表了一个 vector，这边的平方就是指,对每一个 element 都去做平方,然后再开根号,这边开根号指的是对每一个 element,向量裡面的每一个 element,都去做开根号,得到 σ\sigmaσ,反正你知道我的意思就好

接下来就是将每一个zzz都去减掉μμμ 除以 σ\sigmaσ,你把 ziz^izi减掉 μμμ,除以 σ\sigmaσ,就得到 ziz^izi的 tilde

所以我们就是把 z1z^1z1 减 μμμ 除以 σ\sigmaσ,得到 z1z^1z1 tilde,同理 z2z^2z2 减 μμμ 除以 σ\sigmaσ,得到 z2z^2z2 tilde,z3z^3z3 减 μμμ 除以 σ\sigmaσ,得到 z3z^3z3 tilde,那就把这个 z1z^1z1 z2z^2z2 z3z^3z3,做 Feature Normalization,变成 z1z^1z1 tilde,z2z^2z2 tilde 跟 z3z^3z3 的 tilde。

注意

这边的μ\muμ跟σ\sigmaσ其实都是根据z1z2z3z^1 z^2 z^3z1z2z3计算出来的

所以这边 z1z^1z1 啊,它本来,如果我们没有做 Feature Normalization 的时候,你改变了 z1z^1z1 的值,你会改变这边 a 的值,但是现在啊,当你改变 z1z^1z1 的值的时候,μμμ 跟 σ\sigmaσ 也会跟著改变,μμμ 跟 σ\sigmaσ 改变以后,z2z^2z2 的值 a2a^2a2 的值,z3z^3z3 的值 a3a^3a3 的值,也会跟著改变。

所以之前，我们每一个x1~x2~x3~\tilde{x_1} \tilde{x_2} \tilde{x_3}x1​~​x2​~​x3​~​是分开处理的，但是在做了Featrue Normalization以后，这三个example就变得彼此关联了。

我们这边z1z^1z1只要有改变，接下来z2a2z3a3z^2 a^2 z^3 a^3z2a2z3a3也都会跟着改变，所以这边其实只要我们有做Feature Normalization的时候，就要把一整个的process，就是有收集一堆feature，把这堆feature算出μ\muμ跟σ\sigmaσ这件事，也当作是network的整体的一部分。

也就是说，我们现在有一个比较大的network

  • 之前的network：进一个input得到一个output
  • 现在的大的network：先是进一堆input，用这堆input在这个network里面，要算出μ\muμ和σ\sigmaσ然后产生一堆output

这个地方比较抽象，只可意会不可言传

现在其实有一个很明显问题，我们的训练资料往往data很多，GPU的内存memory根本没有办法直接让整个data set的data都load进去

在实作的时候，你不会让一个network考虑整个training data里面有所的example，你只会考虑一个batch里面的example，举例说，batch设64，那你这个巨大的network就把64笔data读进去，算除这64笔的μ\muμ和σ\sigmaσ，对着64笔data都去做normalization，因为我们在实作的时候，我们只对一个batch里面data，做normalization，这招叫做Batch Normalization

Batch Normalization

有一个问题就是说，我们想算μ\muμ和σ\sigmaσ，但是batch只有1的大小，这就根本算不出来。所以这个适用于batch size比较大的时候，因为batch size比较大，也许这个batchsize里的data就足以表示整个分布，那么这个时候，就可以把本来对整个corpus做Feature的Normalization这件事情，改成只在一个batch，做Feature Normalization，作为估计。

在做Batch Normalization的时候，往往还需要我们算出这个z~\tilde{z}z~之后

  • 再乘上另一个向量叫做γ\gammaγ，这个γ\gammaγ是一个vector，可以直接让z~\tilde{z}z~和γ\gammaγ做element wise的相乘，把这个向量里面的element，跟γ\gammaγ这个向量里面的element，两两相乘
  • 再加上β\betaβ这个向量得到z~\tilde{z}z~

而 βββ 跟 γγγ,你要把它想成是 network 的参数,它是另外再被learn出来的

为什么要加上 βββ 跟 γγγ呢？

有人可能会觉得说，如果我们做normalization以后，那这边的z~\tilde{z}z~的平均就一定是0，那么或许平均是0的话，就是给那个network一些新的限制了，也许这种限制会带来什么负面的影响，所以我们把β\betaβ和γ\gammaγ加回去

然后让network现在的hidden layer的output平均不是0的话，它就自己去learnβ\betaβ和γ\gammaγ调整一下z~\tilde{z}z~的分布

但这边我们想，刚才不是说Batch Normalization就是为了让每一维的range都是一样的吗，现在如果加去乘上γ\gammaγ，再加上β\betaβ这样调整了之后range不久又不一样了吗？

是有可能的。但是你实际上在训练的时候,这个 γγγ 跟 βββ 的初始值

  • 会把这个 γγγ 的初始值 就都设為 1,所以 γγγ 是一个裡面的值,一开始其实是一个裡面的值,全部都是 1 的向量

  • βββ 是一个裡面的值,全部都是 0 的向量,所以 γγγ 是一个 one vector,都是 1 的向量,βββ 是一个 zero vector,裡面的值都是 0 的向量

    所以让你的network在一开始训练的时候，每一个dimension的分布是很接近的，也许训练到一个比较好的error surface了，走到一个比较好的地方之后，再把γ\gammaγ和β\betaβ慢慢加进去。

以上说的都是training的部分，testing，有时候又叫inference

Testing

这个Batch Normalization再Testing的时候会出现什么问题呢？

在 testing 的时候,如果 当然如果今天你是在做作业,我们一次会把所有的 testing 的资料给你,所以你确实也可以在 testing 的资料上面,製造一个一个 batch

假设真的有系统上线了，是一个真正的app，我们不能说必须等到64个batch或者其他规定好的batch都进来才做运算

但是在做Batch Normalization的时候一个x~\tilde{x}x~，一个normalization过的feature进来，然后有一个zzz，这个zzz要先减去μ\muμ除去σ\sigmaσ，这个μ\muμ和σ\sigmaσ是用一个batch的资料算出来的。

如果今天根本没有batch怎么办？

在pytorch中，如果是做Batch Normalization的话，在testing的时候，并不需要做什么特别的处理，Pytorch已经帮助我们处理好了。

在training的时候，如果你有在做 Batch Normalization 的话,在 training 的时候,你每一个 batch 计算出来的 μμμ 跟 σ\sigmaσ,他都会拿出来算 moving average

training每次取出一个batch的时候，都会算一个μ1\mu^1μ1，取第二个batch出来的时候，你就算个μ2\mu^2μ2一直取到第ttt个batch出来的时候，就算一个μt\mu^tμt

接下来你会算一个 moving average,你会把你现在算出来的 μμμ 的一个平均值,叫做 μμμ bar,乘上某一个 factor,那这也是一个常数,这个这也是一个 constant,这也是一个那个 hyper parameter,也是需要调的那种

Pytorch里面的这个参数设为了0.1，我记得他 P 就设 0.1,好,然后加上 1 减 P,乘上 μtμ^tμt ,然后来更新你的 μμμ 的平均值,然后最后在 testing 的时候,你就不用算 batch 裡面的 μμμ 跟 σ\sigmaσ 了

因为testing的时候，在真正application上，没有batch这个东西，就直接拿 μˉ\barμμˉ​ 跟 σˉ\bar\sigmaσˉ ,也就是 μμμ 跟 σ\sigmaσ 在训练的时候,得到的 moving average,μˉ\barμμˉ​ 跟 σˉ\bar\sigmaσˉ ,来取代这边的 μμμ 跟 σ\sigmaσ,这个就是 Batch Normalization,在 testing 的时候的运作方式

Internal Covariate Shift?

好 接下来的问题就是,Batch Normalization,它為什麼会有帮助呢,在原始的 Batch Normalization,那篇 paper 裡面,他提出来一个概念,叫做 internal covariate shift,covariate shift（训练集和预测集样本分布不一致的问题就叫做“covariate shift”现象) 这个词汇是原来就有的,internal covariate shift,我认為是,Batch Normalization 的作者自己发明的

他认為说今天在 train network 的时候,会有以下这个问题,这个问题是这样

network 有很多层

  • x 通过第一层以后 得到 a

  • a 通过第二层以后 得到 b

  • 计算出 gradient 以后,把 A update 成 A′,把 B 这一层的参数 update 成 B′

但是作者认為说,我们在计算 B,update 到 B′ 的 gradient 的时候,这个时候前一层的参数是 A 啊,或者是前一层的 output 是小 a 啊

那当前一层从 A 变成 A′ 的时候,它的 output 就从小 a 变成小 a′ 啊

但是我们计算这个 gradient 的时候,我们是根据这个 a 算出来的啊,所以这个 update 的方向,也许它适合用在 a 上,但不适合用在 a′ 上面

如果说Batch Normalization的话，我们会让aaa和a′a'a′的分布比较接近，可能就说这样它帮助了训练

但是！How Does Batch Normalization Help Optimization 这篇paper打脸了internal covariate shift这个观点。

这篇paper里面，它从多个方面这个internal covariate shift这个观点首先它不是一个training network的问题，这个Batch normalization训练结果比较好可能并不见得因为解决了这个问题。

那么为什么说Batch Normalization之后会比较好呢，在这篇How Does Batch Normalization Help Optimization这篇论文里面，从实验上支持了Batch Normalization可以改变error surface使得更加平坦这个观点。而且其实不见得非得做Batch Normalization去优化error surface，很多其他的方法也可以完成。

所以这个方法只是一个偶然，虽然它确实有用。
